## 第 2 页 · 先说清楚：QQ 更像“平台接入项目”

### 这一页和 Telegram / Discord 的差别

QQ 通常不是那种“装完就能直接 `channels add --channel xxx`”的纯开箱渠道，它往往还会涉及：

- 开放平台资格 / 实名 / 审核
- Webhook 或 WebSocket 接入方式
- 课程配套插件、企业私有适配器，或二次封装通道

### 所以这页课的重点不是死记字段

而是掌握一套能复用到客户现场的闭环：

```text
平台凭证 → 适配器安装/配置 → config validate → gateway restart → channels status --probe → 实际群聊验证
```

> ⚠️ 如果你用的是课程配套或客户私有 QQ 适配器，最终字段名以那份 README 为准，不要死套某一个示例 JSON。

> 配置没过、网关没重启、渠道没探测之前，不要急着去群里 @ 机器人。

## 第 3.5 页 · 讲师串场话术

- **开场认知**：QQ 不是“更难的 Telegram”，它更像一个有审核、有平台规则、有交付约束的国内项目。
- **实操前提醒**：这一节最容易卡在审核、配置口径和平台限制，不要让学员误以为是 OpenClaw 本身不稳定。
- **卡点转场**：如果机器人不回，不要先改 Prompt，先确认审核状态、可见范围、Webhook/WebSocket 模式和渠道探测结果。
- **复盘收口**：QQ Lab 的价值，不只是接一个机器人，而是让学员感受到“平台型接入项目”和“纯技术跑通”之间的差别。

## 第 4 页 · 第一步：在 QQ 开放平台创建应用

### 操作步骤

1. 打开 [QQ 开放平台](https://q.qq.com)
2. 完成开发者实名认证
3. 创建应用（QQ 群机器人 / QQ 频道机器人，按你的场景选）
4. 记录平台给你的关键凭证：
   - `App ID`
   - `App Secret`
   - `Token`
   - 如适配器要求，可能还有回调地址、沙箱信息等
5. 提交审核

### 一句实话

QQ 最大的不确定性通常不在 OpenClaw，而在：

- 平台审核
- 事件订阅是否可达
- 你拿到的适配器到底期望什么字段

---

## 第 5 页 · 第二步：把 QQ 适配器接进 OpenClaw

### 推荐顺序

1. **先看适配器 README**
2. 如果适配器支持 `openclaw channels add`，优先走 CLI
3. 如果不支持，再写 `channels.<channel-id>` 配置

### 一个常见的配置示意

```json5
{
  channels: {
    qq: {
      enabled: true,
      appId: "你的App ID",
      appSecret: "你的App Secret",
      token: "你的Token"
    }
  }
}
```

> ⚠️ 这里只是**示意结构**。不同实现里，真正的渠道 id 可能是 `qq`、`qqguild`、`qq-channel` 或者别的名字。

### 配完以后马上做三件事

```bash
openclaw config validate
openclaw gateway restart
openclaw channels status --probe
```

## 第 6 页 · 第三步：在 QQ 里验证真实消息链路

### 测试步骤

1. 把机器人加入一个你可控的测试群 / 频道
2. 先发一条最简单的消息，比如：`@AI助手 你好`
3. 观察是否收到回复
4. 再试第二条、第三条，确认是否有上下文
5. 最后再测一个真实任务：`@AI助手 帮我整理一下今天的待办`

### ✅ 成功标志

- 群里能稳定触发回复
- 第二轮消息不是“失忆式回复”
- 延迟可接受

> 先在测试群验证，再上正式群。不要把正式群当调试环境。

## 第 7 页 · 故障排查速查

| 问题 | 先查什么 |
|------|----------|
| 审核迟迟不过 | 平台侧问题，先看开放平台反馈，不要先改 OpenClaw |
| 群里完全不回复 | `channels status --probe`、回调可达性、适配器是否已加载 |
| 配置刚改完完全失效 | 有没有先跑 `openclaw config validate`，有没有 `openclaw gateway restart` |
| 机器人偶尔在线偶尔离线 | WebSocket / 回调链路是否稳定；防火墙是否拦截 |
| 回复被平台拦截 | 内容安全限制；在 `SOUL.md` 里加更保守的回复边界 |
| 字段都填了还是报错 | 大概率是字段名与适配器要求不一致，回 README 核对 |

### 一个很重要的交付习惯

把你最后确认有效的：

- 适配器名称
- 渠道 id
- 凭证字段名
- 回调地址

记录在项目文档里。QQ 最怕“第二个人接手时不知道你到底接的是哪一套实现”。
