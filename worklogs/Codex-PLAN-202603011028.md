# 工作日志与项目近况方案（纳入 `openclaw-feishu`）

## 摘要
本方案将把项目近况统计口径从“仅主仓”升级为“`openclaw-course + openclaw-feishu` 联合统计”，并继续以 [worklogs/codex-work-logs.md](/e:/openclaw-course/worklogs/codex-work-logs.md) 作为唯一日志主档。
已确认的固定规则如下：北京时间 `YYYY-MM-DD HH:mm`、每次交付写 1 条、单文件逆序追加、必须记录任务指派/处理过程/交付成果。
“最近 14 批提交”将按两仓时间合并后取最新 14 条。

## 范围与成功标准
1. 规范范围：日志治理仅覆盖 [worklogs/codex-work-logs.md](/e:/openclaw-course/worklogs/codex-work-logs.md) 的结构与记录规则，不改业务代码。
2. 近况范围：默认统计仓库为 `e:\openclaw-course` 与 `e:\openclaw-course\openclaw-feishu`。
3. 成功标准：
1. 日志中每条记录均含分钟级时间戳（北京时间）。
2. 日志中明确标注提交所属仓库。
3. 近况区块能稳定产出“联合最近 14 条提交”摘要。
4. 当某仓无足够提交时，日志中会写明补位策略与统计来源。

## 公共接口/类型变更（文档接口）
将日志条目 Schema 固化为以下字段（新增 `仓库范围` 与 `提交来源` 字段）：

```md
## [提交时间 YYYY-MM-DD HH:mm] <任务标题> [TASK-YYYYMMDD-序号]

- 任务来源:
- 仓库范围: openclaw-course | openclaw-feishu | both
- 指派时间:
- 开始时间:
- 提交时间:
- 任务目标:
- 执行过程:
  1) ...
  2) ...
- 交付成果:
  1) ...
  2) ...
- 变更清单:
- 提交来源:
  - repo: ...
  - branch: ...
  - head: ...
  - ahead/behind: ...
- 验证结果:
- 后续建议:
```

## 实施步骤（决策完成版）
1. 在 [worklogs/codex-work-logs.md](/e:/openclaw-course/worklogs/codex-work-logs.md) 顶部写入“日志准则固定区”，明确：
1. 架构型、技术型、记录型文档输出到 `worklogs/`。
2. 统一北京时间到分钟。
3. 每次交付 1 条完整记录。
4. 默认联合统计 `openclaw-course + openclaw-feishu`。
2. 新增“任务编号规则”：`TASK-YYYYMMDD-序号`，按当日递增。
3. 写入本次任务首条记录（你本轮指派 + 我的处理过程 + 当前交付）。
4. 在日志中新增“项目近况基线（最近 14 批）”区块，使用两仓合并排序结果。
5. 在近况区块固定输出 4 个维度：
1. 时间窗口（起止时间）。
2. 两仓占比（每仓贡献条数）。
3. 主题聚类（课程、guard、feishu 同步/定制等）。
4. 风险与待办（例如 ahead 未推送、未跟踪文件）。
6. 建立后续更新机制：每次你下发任务先记“起始时间+目标”，每次我交付时补“提交时间+结果+验证”。

## 最近 14 批提交口径（已锁定）
1. 统计方法：两仓 `git log` 合并，按提交时间降序取 14。
2. 当前可得结果：最近 14 条由 `openclaw-course` 与 `openclaw-feishu` 共同构成。
3. 当前构成（本轮扫描结果）：
1. `openclaw-course` 贡献 10 条（2026-02-22 至 2026-02-26）。
2. `openclaw-feishu` 贡献 4 条（2026-02-10 至 2026-02-26，其中包含 2026-02-26、2026-02-23 的上游合并提交）。
4. 默认展示粒度：提交级摘要（hash、时间、仓库、主题），不展开逐文件 diff。

## 测试用例与验收场景
1. 字段完整性测试：随机抽 3 条日志，必须都含“指派/开始/提交时间 + 仓库范围 + 提交来源”。
2. 时间格式测试：全文件匹配 `YYYY-MM-DD HH:mm`，且统一按北京时间书写。
3. 逆序测试：最新提交记录必须位于最上方。
4. 跨仓统计测试：同一份近况区块能看到两仓来源，不得只出现单仓。
5. 容错测试：当 `openclaw-feishu` 不可访问时，日志记录“降级为单仓统计 + 原因 + 时间”。
6. 一致性测试：近况中的“提交条数/时间窗口”与实际 `git log` 查询一致。

## 假设与默认值
1. 你后续所说“项目近况”默认指两仓联合：`openclaw-course + openclaw-feishu`。
2. 你后续若未特别说明，不纳入 `openclaw` 官方仓统计。
3. 日志主文件固定为 [worklogs/codex-work-logs.md](/e:/openclaw-course/worklogs/codex-work-logs.md)，不拆分子文件。
4. 本方案仅定义并落地记录机制，不涉及代码功能实现与发布动作。
