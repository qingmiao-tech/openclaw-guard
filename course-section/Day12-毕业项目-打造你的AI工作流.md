## 第 3 页 · 什么是 AI 工作流？

### 定义

将多个 AI 能力**串联**起来，形成一个自动化的工作管线

### 工作流 vs 单一任务 vs AI 团队

| 对比项 | 单一任务 | 工作流（本节） | AI 团队（Day 13） |
|--------|----------|----------------|-------------------|
| Agent 数量 | 1 个 | 1 个 | 多个 |
| 范围 | 让 AI 做一件事 | 让 AI 自动完成一系列事 | 多个 AI 各司其职 |
| 触发 | 手动触发 | 自动触发 + 手动触发 | 多渠道独立触发 |
| 能力 | 单一能力 | 多能力串联 | 多角色协作 |
| 价值 | 节省一次时间 | 持续节省时间 | 替代一个团队 |

> 💡 Day 12 的重点是**一个 Agent 串联多种能力**形成工作流。
> 如果你想让多个 Agent 扮演不同角色、各自独立工作甚至互相协作，那是 Day 13 的内容。

### 好的工作流特征

- ✅ 解决真实痛点
- ✅ 流程清晰
- ✅ 容错性强
- ✅ 可持续运行

## 第 5 页 · 选题自检

### 三个问题

在确定选题前，问自己：

1. ❓ 这个任务是否**重复性高**？
2. ❓ 是否有**明确的输入输出**？
3. ❓ 能否用 OpenClaw **现有能力**实现？

### 一句话描述

> "我要做一个 ___，它能自动 ___"

例如：
- "我要做一个**行业资讯聚合器**，它能自动**每天收集 AI 新闻并推送到 Telegram**"
- "我要做一个**学习助手**，它能自动**每天推送英语单词并在晚上复习**"

## 第 6 页 · 项目设计方法论

### 四步法

```
第一步：需求分析
  → 明确问题、定义输入输出、列出所需能力
      ↓
第二步：流程设计
  → 画出流程图、确定每步工具、设计异常处理
      ↓
第三步：分步实现
  → 先跑通主路径、再加异常处理、最后配置自动化
      ↓
第四步：测试验证
  → 手动测试、端到端测试、异常场景测试
```

## 第 6.5 页 · 毕业项目方案检查表：别一上来就做大而全

毕业项目最容易犯的错不是“不会做”，而是“做太大”。

建议开做前，先用这张表把方案收紧：

| 问题 | 你的答案 |
|------|------|
| 这个项目的触发方式是什么？ | 手动 / Cron / Heartbeat |
| 它是否真的需要 Browser？ | 如果不用浏览器也能做，就先别上 |
| 它是否真的需要 Memory？ | 如果只是一锤子任务，先别把记忆系统复杂化 |
| 它是否真的需要 Channel 推送？ | 如果课堂先验收，先本地文件输出也可以 |
| 它是否真的需要多个 Agent？ | 不需要就留到 Day 13，再做多 Agent 版 |

一句话建议：

> **毕业项目先做“最小可运行版本”，而不是“终极完整版”。**

## 第 7 页 · 实战示例：每日资讯聚合推送

### 需求

每天早上 9 点自动收集 AI 热点新闻，推送到 Telegram

### 流程设计

```
Cron 定时触发 (09:00)
    ↓
调用"新闻收集" Skill
    ↓
Web 搜索获取最新资讯
    ↓
AI 筛选和摘要
    ↓
保存为本地文件（备份）
    ↓
通过 Telegram 推送给用户
```

### 涉及能力

Cron + Skill + Web 搜索 + Memory + 渠道推送

---

## 第 7.5 页 · 项目实施顺序：先手动跑通，再自动化

这一页建议作为讲师现场不断提醒的操作纪律：

1. **先手动跑通主路径**
   不上 Cron，不上复杂推送，先确认核心 Skill 能产出结果。
2. **再补输入输出文件**
   让工作流至少能把结果写到本地文件，形成可见产物。
3. **再补渠道推送或定时触发**
   只有主路径稳定后，才加 Telegram / 飞书 / Cron。
4. **最后补异常处理和回退**
   不是最后才想排障，而是最后才把排障写成标准动作。

这能显著减少“到底是 Skill 坏了、Cron 坏了，还是渠道坏了”的混乱。

## 第 10 页 · 实操任务 2：分步实现

### 操作步骤

1. **实现核心 Skill**：编写并测试项目需要的 Skill
2. **配置基础设施**：设置 Cron、配置渠道推送
3. **串联工作流**：手动触发测试完整流程
4. **逐步调试**：
   - 先测试每个步骤独立工作
   - 再测试步骤间的衔接
   - 最后测试端到端流程
5. **做上线前检查**：
   - `openclaw config validate`
   - `openclaw gateway health`
   - 如有渠道：`openclaw channels status --probe`
6. **配置自动化**：确认手动测试通过后，配置 Cron
7. **观察运行**：让工作流自动运行 1-2 次

### ✅ 成功标志

- 工作流可自动运行并产出预期结果

## 第 10.5 页 · 毕业项目最小交付包

建议每位学员毕业时至少交这 5 样，不求大，但求完整：

1. **一句话项目定义**
   “我要做一个 ___，它能自动 ___。”
2. **一页流程说明**
   列出输入、处理、输出、触发方式。
3. **可运行的核心文件**
   至少包含 `SKILL.md`、必要配置、输出目录。
4. **一份验收证据**
   可以是运行截图、输出文件、消息推送结果。
5. **一份回退说明**
   配置改坏时怎么恢复，至少写出 3 步。

这样你交上来的就不是“一个想法”，而是一套能复跑的作品。

3. 你如何确认问题出在 Skill、渠道，还是 Cron？

## 第 12 页 · 故障排查速查

| 问题 | 解决方案 |
|------|----------|
| 不知道选什么题目 | 想想每天花时间最多的重复工作；问 AI 推荐 |
| 某个步骤执行失败 | 在 Skill 中加入错误处理；查看日志定位 |
| 工作流输出质量不稳定 | 优化 Skill 指令；利用 Memory 积累经验；先固定输入模板 |
| 不知道如何评估效果 | 从"是否减少了重复工作"角度出发 |
| 自动化一加就乱了 | 回到“先手动跑通，再自动化”的顺序，先关掉 Cron 验主路径 |

### 统一排查顺序（课堂建议背下来）

1. `openclaw config validate`
2. `openclaw gateway health`
3. 如有渠道：`openclaw channels status --probe`
4. 手动触发一次主路径
5. 最后再检查 Cron / Heartbeat / 渠道路由

## 第 13 页 · 🔒 安全提示

- **工作流权限最小化**：只给必要的权限
- **定时任务频率控制**：避免过于频繁，浪费 API 额度
- **敏感数据处理**：确保数据不会被意外泄露
- **毕业项目先保守上线**：第一版先保证稳定，再逐步加浏览器、推送、自动化频率
