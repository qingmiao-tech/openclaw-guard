# 渠道与模型

这两类能力决定了 Guard 能不能真正跑起来。

## 先配模型，再接渠道

推荐顺序：

1. 先在“模型”页配置 Provider 和主模型
2. 再在“渠道”页接入至少一个官方渠道

原因很简单：

- 没有主模型，系统的默认行为不稳定
- 没有渠道，系统就算能跑，也没有真实消息入口

## 模型页主要做什么

- 配置 Provider
- 设定主模型
- 设定回退模型链

推荐先把“能稳定工作”放在第一位，而不是一上来就堆很多模型。

## 渠道页为什么不单独拆“飞书”

公开首发阶段，Guard 优先保留通用性。

所以渠道页当前只强调官方渠道入口，不把某个项目定制插件单独放成顶级菜单。这样做的好处是：

- 公开版更容易理解
- 其他用户不会因为缺少某个自定义插件而困惑
- 后续要扩展插件体系时，也能以“官方渠道 + 自定义扩展”的方式自然演进

## 推荐实践

- 模型页至少保证有一个主模型
- 渠道页至少接通一个真实消息渠道
- 如果需要更复杂路由，先保证基础链路可用，再慢慢加备用模型和更多渠道
