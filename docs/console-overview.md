# 控制台导览

Guard 的控制台不是“功能总表”，而是一个带路页加若干关键页面的组合。

## 首页先看什么

首页优先展示三类信息：

- 当前可用性：Guard Web、Gateway、OpenClaw、模型、渠道、备份状态
- 建议动作：当前最值得先做的 4 条主路径
- 风险提示：安全失败项、服务未运行、恢复点缺失、未读提醒等

这意味着首次进入时，你不需要自己猜“先点哪个菜单”。

## 核心控制台

### 首页

用来判断“现在能不能用、下一步先做什么”。

### 运维

用来处理：

- Guard Web 状态
- Gateway 启停与重启
- 运行路径
- 本机环境
- 服务层面的排障

### OpenClaw

用来做：

- 安装 / 修复
- 更新到推荐版本
- 指定版本或回退
- 卸载
- 查看安装来源与生命周期状态

### 渠道

统一管理官方渠道，而不是把某一个渠道独立做成顶级页签。

### 模型

配置 Provider、主模型和回退模型。产品语言上更偏“模型配置”，而不是泛泛的“AI 设置”。

### 安全

把安全检查、权限模式和主机加固放到一个语义闭环里。

### 备份与恢复

默认进入恢复中心，先讲“保存现在 / 恢复到某个状态 / 恢复后继续往前走”；高级 Git 细节留在页内专家视图。

## 工作区与角色

这一组更适合日常内容管理：

- 角色
- 文件
- 搜索

这里的重点不是“展示所有内部状态”，而是帮助你理解工作区和长期资产是否已经准备好。

## 运行与排查

这一组偏观察与诊断：

- 会话
- 日志
- 通知

普通用户平时不一定需要常驻使用，但一旦出现异常，这些页面会很重要。

## 自动化

目前只保留 `Cron` 作为单独能力入口。首发阶段不会把更多自动化概念放到首页主路径中。

## 推荐使用顺序

### 第一次使用

1. 首页
2. OpenClaw
3. 模型
4. 渠道
5. 备份与恢复

### 日常使用

1. 首页确认是否有风险
2. 按需进入渠道 / 模型 / 文件
3. 出现异常时再回运维、日志、通知

### 交付或排障

1. 首页看总体状态
2. 运维确认服务层
3. OpenClaw 页面确认安装来源与更新策略
4. 导出诊断包
5. 再决定是否需要恢复或提 Issue
