# OpenClaw 多模型自动切换实战复盘：从“没生效”到官方补丁落地

> 这篇文章记录一次真实排障过程：我们原本只想做多渠道自动切换，结果连续踩到几个关键坑。最后不仅把链路跑通，还补了一处可上游提交的 failover 修复。

## 一、问题背景

目标很明确：

- 接入多个模型渠道（主路由 + 备用路由）
- 当某个渠道不可用、超时、额度异常时自动切换
- 避免机器人在生产群里“直接瘫痪”

实际遇到的故障现象是：

- 会话里持续出现 `Connection error.`
- 但模型没有切换到下一个 fallback

## 二、先讲结论（适合赶时间的同学）

1. **只配置 provider 不等于自动切换生效**，你必须配置 `agents.defaults.model.fallbacks`。
2. `agents.defaults.model.backup` 不是 OpenClaw 标准回退字段，通常不会被模型回退链识别。
3. 配置改了但运行态没重启，会出现“我明明改了配置，却还是老模型”的错觉。
4. 这次的核心 bug 是：`Connection error.` 没被归类为 failover 错误，导致不触发轮转。
5. 官方补丁思路：把连接/网络类错误纳入 `timeout` 分类，从而触发 fallback。

## 三、这次踩过的 3 个大坑

## 坑 1：配置看起来对了，但运行态没生效

我们当时把 primary 改成不可用 provider，用于验证 fallback。配置文件里已经是新值，但会话仍显示旧模型。

根因不是配置没改成功，而是 **Gateway 进程没有重启**，运行态还在用旧加载状态。

经验规则：

- 修改 `openclaw.json` 后，必须执行一次：

```bash
openclaw gateway restart
```

- 再开新会话（如 `/new`）验证，避免旧会话上下文干扰。

## 坑 2：把 `backup` 当成模型回退字段

很多教程会写：

```json
"model": {
  "primary": "primary/xxx",
  "backup": "backup/yyy"
}
```

这在 OpenClaw 里不是标准自动回退写法。正确字段是：

```json
"model": {
  "primary": "primary/xxx",
  "fallbacks": [
    "backup/yyy"
  ]
}
```

换句话说：**不写 `fallbacks`，就不要期待跨模型自动轮转**。

## 坑 3：`Connection error.` 出现后没有自动切换

这是最关键的一次：

- 首候选模型报 `Connection error.`
- 没有切到第二候选

深入看代码后发现：该报错字符串没有命中 failover 分类器。

结果：系统把它当成“非可回退错误”，自然不会触发 fallback 链。

## 四、这次做的“官方补丁”是什么

本次补丁核心是：**把连接类错误统一归到 `timeout`**，让它走可回退路径。

补丁覆盖了两层：

1. `pi-embedded-helpers/errors.ts`
- 扩展 `ERROR_PATTERNS.timeout`
- 新增关键词：
  - `connection error`
  - `network error`
  - `network request failed`
  - `fetch failed`
  - `socket hang up`
- 新增正则：
  - `ECONNREFUSED / ECONNRESET / ECONNABORTED`
  - `ENOTFOUND`
  - `EAI_AGAIN`

2. `failover-error.ts`
- 扩展 `TIMEOUT_HINT_RE`
- 保证非 assistant 直抛错误也能识别为 timeout 并参与回退

同时补了对应单测样例（连接失败、DNS 失败、fetch 失败等）。

## 五、推荐的稳定配置模板（文本模型）

```json
{
  "agents": {
    "defaults": {
      "model": {
        "primary": "openai-codex/gpt-5.3-codex",
        "fallbacks": [
          "wenwen/claude-opus-4-6",
          "custom-127-0-0-1-11434/qwen3:8b"
        ]
      }
    }
  }
}
```

说明：

- fallback 顺序建议按“稳定性优先”排序，不要把已知不稳定通道放在首位。
- 每次改链路后都跑一次探针：

```bash
openclaw models status --probe --json
```

## 六、生产验证 checklist

1. `openclaw models status --probe --json` 中 fallback 链顺序正确。
2. 首模型故障注入后，日志可见自动尝试下一候选。
3. 用新会话验证，不使用旧会话历史。
4. Gateway 重启时间晚于配置修改时间。
5. 避免在业务高峰直接做故障注入。

## 七、为什么这次复盘值得写

很多“自动切换没生效”不是一个点的问题，而是三层叠加：

- 配置层：字段写错（`backup` vs `fallbacks`）
- 运行层：进程未重启（配置未热生效）
- 代码层：错误分类遗漏（`Connection error.` 没进 failover）

单看任何一层都像“偶发问题”，叠加后就是“系统看起来很玄学”。

这次把三层同时打通后，模型切换行为就稳定可预测了。

## 八、后续建议

1. 将“模型配置变更后必须重启 + 新会话验证”写进团队 SOP。
2. 把 `models status --probe --json` 纳入巡检任务。
3. 继续补齐异常语义映射（例如更多网关/代理厂商错误文本）。
4. 若你在维护分叉版本，建议尽快合并这类 failover 分类补丁。

---

如果你也在做 OpenClaw 多渠道接入，这篇文章里的坑基本都是高概率会踩到的。先把字段、重启、错误分类这三件事做对，自动切换才会真正可靠。
