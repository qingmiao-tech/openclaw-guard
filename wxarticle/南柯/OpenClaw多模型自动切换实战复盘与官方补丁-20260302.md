# OpenClaw 多模型自动切换实战复盘：从”没生效”到官方补丁落地

> **更新（2026-03-03）**：本文提到的官方补丁已被 OpenClaw 主仓库采纳并合并（[PR #31697](https://github.com/openclaw/openclaw/pull/31697)），提交哈希 `30ab9b2`。如果你使用的是 OpenClaw 2026.3.3+ 版本，连接错误自动切换功能已经内置。

## 写在前面：为什么现在必须配置多渠道

近期 AWS 对 API 滥用管控趋严，市面上大量通过 Kiro 反代提供 Claude API 的中转服务变得极不稳定：

- 某些中转今天能用，明天就 `Connection error.`
- 部分服务商直接跑路或关停
- 即使是付费渠道，也可能因上游限流导致间歇性不可用

**如果你希望 OpenClaw 机器人在生产环境稳定运行**，单一渠道已经不够了。多模型自动切换从”锦上添花”变成了”刚需”。

这篇文章记录一次真实排障过程：我们原本只想做多渠道自动切换，结果连续踩到几个关键坑。最后不仅把链路跑通，还发现并修复了一处上游 bug，**并成功提交 PR 被官方采纳**。

## 一、问题背景

目标很明确：

- 接入多个模型渠道（主路由 + 备用路由 + 本地兜底）
- 当某个渠道不可用、超时、额度异常时自动切换
- 避免机器人在生产群里”直接瘫痪”

实际遇到的故障现象是：

- 会话里持续出现 `Connection error.`
- 但模型没有切换到下一个 fallback
- 配置看起来完全正确，但就是不生效

## 二、先讲结论（适合赶时间的同学）

1. **只配置 provider 不等于自动切换生效**，你必须配置 `agents.defaults.model.fallbacks`。
2. `agents.defaults.model.backup` 不是 OpenClaw 标准回退字段，通常不会被模型回退链识别。
3. 配置改了但运行态没重启，会出现”我明明改了配置，却还是老模型”的错觉。
4. 这次的核心 bug 是：`Connection error.` 没被归类为 failover 错误，导致不触发轮转。
5. 官方补丁思路：把连接/网络类错误纳入 `timeout` 分类，从而触发 fallback。
6. **补丁已被官方采纳**：如果你使用 OpenClaw 2026.3.3+，无需手动打补丁。

## 三、这次踩过的 3 个大坑

## 坑 1：配置看起来对了，但运行态没生效

我们当时把 primary 改成不可用 provider，用于验证 fallback。配置文件里已经是新值，但会话仍显示旧模型。

根因不是配置没改成功，而是 **Gateway 进程没有重启**，运行态还在用旧加载状态。

经验规则：

- 修改 `openclaw.json` 后，必须执行一次：

```bash
openclaw gateway restart
```

- 再开新会话（如 `/new`）验证，避免旧会话上下文干扰。

## 坑 2：把 `backup` 当成模型回退字段

很多教程会写：

```json
"model": {
  "primary": "primary/xxx",
  "backup": "backup/yyy"
}
```

这在 OpenClaw 里不是标准自动回退写法。正确字段是：

```json
"model": {
  "primary": "primary/xxx",
  "fallbacks": [
    "backup/yyy"
  ]
}
```

换句话说：**不写 `fallbacks`，就不要期待跨模型自动轮转**。

## 坑 3：`Connection error.` 出现后没有自动切换

这是最关键的一次：

- 首候选模型报 `Connection error.`
- 没有切到第二候选

深入看代码后发现：该报错字符串没有命中 failover 分类器。

结果：系统把它当成“非可回退错误”，自然不会触发 fallback 链。

## 四、这次做的”官方补丁”是什么

本次补丁核心是：**把连接类错误统一归到 `timeout`**，让它走可回退路径。

### 补丁提交历程

在发现这个问题后，我们：

1. **定位根因**：`Connection error.` 等网络错误没有被 failover 分类器识别
2. **编写补丁**：扩展错误模式匹配，增加完整测试用例
3. **提交 PR**：[#31697](https://github.com/openclaw/openclaw/pull/31697) 提交到官方仓库
4. **官方采纳**：2026-03-03 成功合并到主分支（提交 `30ab9b2`）

### 补丁技术细节

补丁覆盖了两层：

1. **`pi-embedded-helpers/errors.ts`**
   - 扩展 `ERROR_PATTERNS.timeout`
   - 新增关键词：
     - `connection error`
     - `network error`
     - `network request failed`
     - `fetch failed`
     - `socket hang up`
   - 新增正则：
     - `/\beconn(?:refused|reset|aborted)\b/i` - 匹配 ECONNREFUSED/ECONNRESET/ECONNABORTED
     - `/\benotfound\b/i` - 匹配 ENOTFOUND（DNS 解析失败）
     - `/\beai_again\b/i` - 匹配 EAI_AGAIN（临时 DNS 失败）

2. **`failover-error.ts`**
   - 扩展 `TIMEOUT_HINT_RE`
   - 保证非 assistant 直抛错误也能识别为 timeout 并参与回退
   - 同时保留了官方新增的 `(?:abort|error)` 模式（合并时的智能融合）

3. **完整测试覆盖**
   - `”Connection error.”` - 通用连接错误
   - `”fetch failed”` - Fetch API 失败
   - `”network error: ECONNREFUSED”` - 连接被拒绝
   - `”dial tcp: lookup api.example.com: no such host (ENOTFOUND)”` - DNS 解析失败
   - `”temporary dns failure EAI_AGAIN”` - 临时 DNS 故障

### 为什么这个补丁重要

在 AWS 管控趋严的当下，中转服务的不稳定性大幅增加。这个补丁让 OpenClaw 能够：

- **自动识别网络层故障**：不再把连接错误当成”不可恢复错误”
- **快速切换到备用渠道**：减少用户等待时间
- **提升生产环境可用性**：即使主渠道挂了，备用渠道能立即顶上

## 五、推荐的稳定配置模板（文本模型）

### 方式一：命令行配置（推荐新手）

OpenClaw 提供了友好的命令行工具来配置多模型，无需手动编辑 JSON 文件。

#### 步骤 1：设置主模型

```bash
# 设置 OpenAI Codex 作为主模型
openclaw models set openai-codex/gpt-5.3-codex
```

#### 步骤 2：添加备用模型

```bash
# 添加 Qwen 云端模型作为第一备用
openclaw models fallbacks add qwen/qwen3.5-plus

# 添加本地 Ollama 模型作为最后兜底
openclaw models fallbacks add ollama/qwen3:8b
```

#### 步骤 3：验证配置

```bash
# 查看当前模型配置
openclaw models list

# 查看 fallback 列表
openclaw models fallbacks list

# 查看所有模型状态
openclaw models status
```

#### 步骤 4：重启 Gateway 使配置生效

```bash
openclaw gateway restart
```

### 方式二：直接编辑配置文件

如果你更喜欢直接编辑配置文件，可以修改 `~/.openclaw/openclaw.json`：

```json
{
  “providers”: {
    “openai-codex”: {
      “type”: “openai”,
      “apiKey”: “sk-proj-xxxxx”,
      “baseURL”: “https://api.openai.com/v1”
    },
    “qwen”: {
      “type”: “openai”,
      “apiKey”: “sk-xxxxx”,
      “baseURL”: “https://dashscope.aliyuncs.com/compatible-mode/v1”
    },
    “ollama”: {
      “type”: “openai”,
      “baseURL”: “http://127.0.0.1:11434/v1”,
      “apiKey”: “ollama”
    }
  },
  “agents”: {
    “defaults”: {
      “model”: {
        “primary”: “openai-codex/gpt-5.3-codex”,
        “fallbacks”: [
          “qwen/qwen3.5-plus”,
          “ollama/qwen3:8b”
        ]
      }
    }
  }
}
```

**配置说明**：

- **openai-codex**：主力模型，性能最强但可能不稳定
- **qwen**：云端备用，国内访问稳定
- **ollama**：本地兜底，完全离线可用

### 完整配置示例（生产环境推荐）

```json
{
  “providers”: {
    “openai-codex”: {
      “type”: “openai”,
      “apiKey”: “${OPENAI_API_KEY}”,
      “baseURL”: “https://api.openai.com/v1”,
      “timeout”: 60000
    },
    “wenwen”: {
      “type”: “anthropic”,
      “apiKey”: “${ANTHROPIC_API_KEY}”,
      “baseURL”: “https://api.wenwen.ai/v1”,
      “timeout”: 60000
    },
    “qwen”: {
      “type”: “openai”,
      “apiKey”: “${QWEN_API_KEY}”,
      “baseURL”: “https://dashscope.aliyuncs.com/compatible-mode/v1”,
      “timeout”: 30000
    },
    “ollama”: {
      “type”: “openai”,
      “baseURL”: “http://127.0.0.1:11434/v1”,
      “apiKey”: “ollama”,
      “timeout”: 120000
    }
  },
  “agents”: {
    “defaults”: {
      “model”: {
        “primary”: “openai-codex/gpt-5.3-codex”,
        “fallbacks”: [
          “wenwen/claude-opus-4-6”,
          “qwen/qwen3.5-plus”,
          “ollama/qwen3:8b”
        ]
      },
      “maxRetries”: 2,
      “retryDelay”: 1000
    }
  }
}
```

**多层级 fallback 策略**：

1. **主力层**：`openai-codex/gpt-5.3-codex` - 最强性能，优先使用
2. **云端备用层**：`wenwen/claude-opus-4-6` - 中转服务，次优选择
3. **国内备用层**：`qwen/qwen3.5-plus` - 国内访问稳定
4. **本地兜底层**：`ollama/qwen3:8b` - 完全离线，保底可用

**Q：配置修改后为什么没生效？**

1. 修改配置文件或执行命令
2. **重启 Gateway**：`openclaw gateway restart`
3. **开新会话**：在聊天中输入 `/new`

说明：

- fallback 顺序建议按”稳定性优先”排序，不要把已知不稳定通道放在首位。
- 每次改链路后都查看一次状态：

```bash
openclaw models status
```

## 六、生产验证 checklist

1. 使用 `openclaw models fallbacks list` 确认 fallback 链顺序正确。
2. 使用 `openclaw models status` 检查所有模型状态。
3. 首模型故障注入后，日志可见自动尝试下一候选。
4. 用新会话验证，不使用旧会话历史。
5. Gateway 重启时间晚于配置修改时间。
6. 避免在业务高峰直接做故障注入。

## 七、为什么这次复盘值得写

很多“自动切换没生效”不是一个点的问题，而是三层叠加：

- 配置层：字段写错（`backup` vs `fallbacks`）
- 运行层：进程未重启（配置未热生效）
- 代码层：错误分类遗漏（`Connection error.` 没进 failover）

单看任何一层都像“偶发问题”，叠加后就是“系统看起来很玄学”。

这次把三层同时打通后，模型切换行为就稳定可预测了。

## 八、后续建议

1. **升级到最新版本**：如果你使用的是 OpenClaw 2026.3.3+，连接错误自动切换已内置，无需手动打补丁。
2. 将”模型配置变更后必须重启 + 新会话验证”写进团队 SOP。
3. 把 `models status --probe --json` 纳入巡检任务。
4. 继续补齐异常语义映射（例如更多网关/代理厂商错误文本）。
5. 若你在维护分叉版本，建议尽快合并这类 failover 分类补丁。
