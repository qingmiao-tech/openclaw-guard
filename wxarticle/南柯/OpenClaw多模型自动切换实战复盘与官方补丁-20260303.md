# OpenClaw 多模型自动切换实战复盘：从”没生效”到官方补丁落地

> **更新（2026-03-03）**：本文提到的官方补丁已被 OpenClaw 主仓库采纳并合并（[PR #31697](https://github.com/openclaw/openclaw/pull/31697)），提交哈希 `30ab9b2`。如果你使用的是 OpenClaw 2026.3.3+ 版本，连接错误自动切换功能已经内置。

## 一、写在前面：为什么现在必须配置多渠道

近期 AWS 对 API 滥用管控趋严，市面上大量通过 Kiro 反代提供 Claude API 的中转服务变得极不稳定：

* 某些中转今天能用，明天就 `Connection error.`
* 部分服务商直接跑路或关停
* 即使是付费渠道，也可能因上游限流导致间歇性不可用

**如果你希望 OpenClaw 机器人在生产环境稳定运行**，单一渠道已经不够了。多模型自动切换从“锦上添花”变成了“刚需”。

这篇文章记录了一次真实的排障过程：我们原本只想做多渠道自动切换，结果连续踩到几个关键坑。最后不仅把链路跑通，还发现并修复了一处上游 bug，**并成功提交 PR 被官方采纳**。

## 二、先讲结论（适合赶时间的同学）

> 💡 **不想看排障过程，只想直接拿作业抄？** 你可以直接向下滚动到 **“五、推荐的稳定配置模板”**，照着里面的命令行或配置文件修改即可。

1. **只配置 provider 不等于自动切换生效**，你必须配置 `agents.defaults.model.fallbacks`。
2. `agents.defaults.model.backup` 不是 OpenClaw 标准回退字段，通常不会被模型回退链识别。
3. 配置改了但运行态没重启，会出现“我明明改了配置，却还是老模型”的错觉。
4. 这次的核心 bug 是：`Connection error.` 没被归类为 failover 错误，导致不触发轮转。
5. 官方补丁思路：把连接/网络类错误纳入 `timeout` 分类，从而触发 fallback。
6. **补丁已被官方采纳**：如果你使用 OpenClaw 2026.3.3+，无需手动打补丁。

## 三、这次踩过的 3 个大坑

### 坑 1：配置看起来对了，但运行态没生效

我们当时把 primary 改成不可用 provider，用于验证 fallback。配置文件里已经是新值，但会话仍显示旧模型。
根因不是配置没改成功，而是 **Gateway 进程没有重启**，运行态还在用旧的加载状态。

**经验规则：**
修改 `openclaw.json` 后，必须执行一次：

```bash
openclaw gateway restart

```

然后再开新会话（如 `/new`）验证，避免旧会话上下文干扰。

### 坑 2：把 `backup` 当成模型回退字段

很多教程会写：

```json
"model": {
  "primary": "primary/xxx",
  "backup": "backup/yyy"
}

```

这在 OpenClaw 里不是标准自动回退写法。正确字段是：

```json
"model": {
  "primary": "primary/xxx",
  "fallbacks": [
    "backup/yyy"
  ]
}

```

换句话说：**不写 `fallbacks`，就不要期待跨模型自动轮转**。

### 坑 3：`Connection error.` 出现后没有自动切换

这是最关键的一次：

* 首候选模型报 `Connection error.`
* 没有切到第二候选

深入看代码后发现：该报错字符串没有命中 failover 分类器。
结果：系统把它当成“非可回退错误”，自然不会触发 fallback 链。

## 四、这次做的“官方补丁”是什么

本次补丁核心是：**把连接类错误统一归到 `timeout**`，让它走可回退路径。

### 补丁提交历程

在发现这个问题后，我们：

1. **定位根因**：`Connection error.` 等网络错误没有被 failover 分类器识别
2. **编写补丁**：扩展错误模式匹配，增加完整测试用例
3. **提交 PR**：[#31697](https://github.com/openclaw/openclaw/pull/31697) 提交到官方仓库
4. **官方采纳**：2026-03-03 成功合并到主分支（提交 `30ab9b2`）

### 补丁技术细节

补丁覆盖了两层：

1. **`pi-embedded-helpers/errors.ts`**
* 扩展 `ERROR_PATTERNS.timeout`
* **新增关键词：** `connection error`、`network error`、`network request failed`、`fetch failed`、`socket hang up`
* **新增正则：** * `/\beconn(?:refused|reset|aborted)\b/i` - 匹配 ECONNREFUSED/ECONNRESET/ECONNABORTED
* `/\benotfound\b/i` - 匹配 ENOTFOUND（DNS 解析失败）
* `/\beai_again\b/i` - 匹配 EAI_AGAIN（临时 DNS 失败）




2. **`failover-error.ts`**
* 扩展 `TIMEOUT_HINT_RE`
* 保证非 assistant 直抛错误也能识别为 timeout 并参与回退
* 同时保留了官方新增的 `(?:abort|error)` 模式（合并时的智能融合）


3. **完整测试覆盖**
* `"Connection error."` - 通用连接错误
* `"fetch failed"` - Fetch API 失败
* `"network error: ECONNREFUSED"` - 连接被拒绝
* `"dial tcp: lookup api.example.com: no such host (ENOTFOUND)"` - DNS 解析失败
* `"temporary dns failure EAI_AGAIN"` - 临时 DNS 故障



### 为什么这个补丁重要

在 AWS 管控趋严的当下，中转服务的不稳定性大幅增加。这个补丁让 OpenClaw 能够：

* **自动识别网络层故障**：不再把连接错误当成“不可恢复错误”
* **快速切换到备用渠道**：减少用户等待时间
* **提升生产环境可用性**：即使主渠道挂了，备用渠道能立即顶上

## 五、推荐的稳定配置模板

⚠️ **核心前提：** 在添加任何 fallback 回退模型之前，请务必确保你已经通过 Auth 授权或配置 API Key 的方式，将对应的模型服务商（Provider）接入了节点中。如果底层的提供商鉴权没打通，哪怕 fallback 列表里写了对应的模型名称，系统在切换时也会因为没有权限而直接调用失败。

为了应对当前复杂多变的网络和管控环境，以下是我目前配置的**四层高可用降级架构**模型链路：

1. **主模型层：** `openai-codex/gpt-5.3-codex` - 日常主力输出，性价比高。
2. **云端备用层：** `wenwen/claude-opus-4-6` - 中转服务，作为主模型不可用时的次优选择。
3. **国内备用层：** `qwen-portal/qwen3.5-plus` - 规避国际网络波动，国内访问极度稳定。
4. **本地兜底层：** `ollama/qwen3:8b` - 完全离线运行，作为断网或上游全挂时的终极保底。

#### 方式一：命令行配置（推荐，不易出错）

OpenClaw 提供了友好的命令行工具来配置多模型。这其实是最不容易出错的方式——程序会自动帮你处理 JSON 格式，避免了因为少写一个逗号导致配置崩溃的尴尬。

**步骤 1：设置主模型**

```bash
# 设置 OpenAI Codex 作为日常主力
openclaw models set openai-codex/gpt-5.3-codex

```

**步骤 2：依次添加备用模型（严格按优先级顺序）**

```bash
# 1. 添加云端中转作为第一备用
openclaw models fallbacks add wenwen/claude-opus-4-6

# 2. 添加国内稳定模型作为第二备用
openclaw models fallbacks add qwen-portal/qwen3.5-plus

# 3. 添加本地完全离线模型作为最后兜底
openclaw models fallbacks add ollama/qwen3:8b

```

**步骤 3：验证配置与重启**

```bash
# 查看 fallback 链路顺序是否符合我们的四层设计
openclaw models fallbacks list

# 查看所有模型可用状态
openclaw models status

# 重启 Gateway 使配置彻底生效
openclaw gateway restart

```

#### 方式二：直接编辑配置文件（适合全局掌控）

如果你更喜欢直接操控底层配置，可以编辑 `~/.openclaw/openclaw.json`。

**注意：** OpenClaw 的提供商（providers）需嵌套在 `models` 节点下，同时别忘了在 `agents.defaults.models` 中配置好模型的映射或别名（alias）。以下是严格对应上述四层架构的完整配置示例：

      "wenwen": {
        "baseUrl": "https://breakout.wenwen-ai.com",

        "models": [

            "name": "Claude Opus 4.6 (Cloud Backup)",
            "api": "anthropic-messages",
            "contextWindow": 200000,
            "maxTokens": 8192
          }
        ]
      },
      "qwen-portal": {
        "baseUrl": "https://dashscope.aliyuncs.com/compatible-mode/v1",
        "apiKey": "${DASHSCOPE_API_KEY}",
        "models": [
          {
            "id": "qwen3.5-plus",
            "name": "Qwen 3.5 Plus (Domestic Backup)",

#### 小白友好提示

**Q：我不知道自己的 API Key 怎么办？**
A：将明文密钥写在配置文件里容易泄露，使用环境变量管理密钥会更安全（就像我们在上面 JSON 里写的 `${WENWEN_API_KEY}` 一样）。

根据你所使用的操作系统，可以参考以下方式配置。下面命令会对应到你在 `~/.openclaw/openclaw.json` 里写的占位符，例如：

```json
"apiKey": "${WENWEN_API_KEY}"
```

**对于 macOS / Linux 用户：**

```bash
# 1) 临时生效（仅当前终端）
export OPENAI_API_KEY="sk-xxx"
export WENWEN_API_KEY="sk-xxx"
export DASHSCOPE_API_KEY="sk-xxx"

# 2) 持久化生效（写入 shell 配置文件）
echo 'export OPENAI_API_KEY="sk-xxx"' >> ~/.bashrc
echo 'export WENWEN_API_KEY="sk-xxx"' >> ~/.bashrc
echo 'export DASHSCOPE_API_KEY="sk-xxx"' >> ~/.bashrc

# zsh 用户请写入 ~/.zshrc
# 让当前窗口立即生效
source ~/.bashrc
```

**对于 Windows 用户（重点：区分“临时生效”和“持久化”）：**

```powershell
# 1) 临时生效（仅当前 PowerShell 会话，关闭窗口即失效）
$env:OPENAI_API_KEY="sk-xxx"
$env:WENWEN_API_KEY="sk-xxx"
$env:DASHSCOPE_API_KEY="sk-xxx"

# 验证当前会话是否已生效
echo $env:OPENAI_API_KEY
echo $env:WENWEN_API_KEY
echo $env:DASHSCOPE_API_KEY
```

```powershell
# 2) 持久化生效（写入“用户环境变量”，新开终端后生效）
setx OPENAI_API_KEY "sk-xxx"
setx WENWEN_API_KEY "sk-xxx"
setx DASHSCOPE_API_KEY "sk-xxx"

# 注意：
# - setx 不会回写当前窗口，需“关闭并重新打开”终端
# - 若 OpenClaw/Gateway 已在运行，重启进程后才会读取新变量
```

*(注：Windows 用户也可以在系统搜索栏输入“环境变量”，通过可视化界面新增上述键值对。)*

**Q：配置修改后为什么没生效？** 记住三步走：

1. 修改配置文件或执行命令。
2. 重启 Gateway：执行 `openclaw gateway restart`。
3. 开新会话：在聊天中输入 `/new`。

## 六、生产验证 Checklist

1. 使用 `openclaw models fallbacks list` 确认 fallback 链顺序正确。
2. 使用 `openclaw models status` 检查所有模型状态。
3. 首模型故障注入后，日志可见自动尝试下一候选。
4. 用新会话验证，不使用旧会话历史。
5. Gateway 重启时间晚于配置修改时间。
6. 避免在业务高峰直接做故障注入。

## 七、为什么这次复盘值得写

很多“自动切换没生效”不是一个点的问题，而是三层叠加：

* **配置层**：字段写错（`backup` vs `fallbacks`）或 Provider 鉴权/结构错误
* **运行层**：进程未重启（配置未热生效）
* **代码层**：错误分类遗漏（`Connection error.` 没进 failover）

单看任何一层都像“偶发问题”，叠加后就是“系统看起来很玄学”。
这次把三层同时打通后，模型切换行为就稳定可预测了。
