# OpenClaw 生命周期

Guard 把 OpenClaw 相关动作收口成一条清晰的生命周期路径，而不是一堆离散按钮。

## 你可以在这里做什么

- 安装 / 修复 OpenClaw
- 查看当前安装来源
- 更新到推荐版本
- 指定版本
- 快速回退
- 卸载 OpenClaw

## 更新策略

Guard 当前采用“官方能力优先，同源更新”的思路：

- 如果当前 OpenClaw CLI 能正常运行，优先沿用它原本的安装来源来完成更新
- 只有在 CLI 缺失、损坏或无法执行官方更新能力时，才回退到 Guard 自己的修复链路

这样做的好处是：

- 用户原本怎么装的，Guard 尽量就怎么升
- 不会轻易把 PATH 上已有的安装强制替换成另一种来源

## 推荐动作只有一个

普通用户默认只需要关心主按钮：

- `更新到推荐版本`

复杂能力会被放进高级区，例如：

- 指定某个版本
- 切换渠道
- 回退到上一版本
- 查看安装来源与诊断细节

## 回退怎么理解

### package 安装

更适合做：

- 回退到上一版本
- 指定某个稳定版本

### git 源码安装

更适合做：

- 切换到稳定 / 测试 / 开发渠道
- 高级用户按 tag / ref / 日期回退

::: warning 高级 git 回退
源码检出场景下的高级回退属于专家能力。它可能让仓库进入 detached HEAD 状态，也要求工作区保持干净。普通用户不建议直接使用。
:::

## 卸载时会发生什么

Guard 已提供“彻底卸载 OpenClaw”的能力，并且要求二次确认。

卸载前建议：

1. 先确认你是否还需要保留当前 OpenClaw 环境
2. 如果不确定，先在“备份与恢复”里保存一个恢复点
3. 如果是客户现场，先导出诊断包再动手

## 推荐实践

- 更新前先创建恢复点
- 看到“有推荐版本”时，优先用主按钮更新
- 只有明确知道自己想回到哪个版本时，再使用指定版本或回退
- 如果出现更新失败，先看安装来源和诊断信息，再决定是修复还是卸载重装
